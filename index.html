The Company

Biomage

Biomage is a bioinformatics company with around 15 employees, specializing in single-cell RNA sequencing. This technology is one of the biggest advancements in biotechnology, and allows biologists to study the genetic profile of tens of thousands of unique cells. However, the computational requirements of such a task are massive, and biologists and other researchers are not equipped to extract insight from a massive data set like this. Data analysis is done by bioinformaticians, who are expensive and each request of a researcher might take days to be fulfilled.

Cellscope by Biomage is a novel product commissioned by Harvard Medical School which allows biologists to upload, process, analyze and publish their data from a browser. As one of the people hired by HMS, my job was to design and build the product.

Problem

Cellscope is a particularly expansive cloud-native project. It is deployed on Amazon Web Services, and contains numerous microservices, static infrastructure such as S3 Buckets, AWS Step Functions, Redis caches, as well as workers which run R code that perform machine learning and bioinformatics tasks. Biomage embraces agile software development and wants to implement continuous delivery with safe and fast release cycles for Cellscope. However, testing is proving to be challenging for the following reasons:

Distributed

The distributed nature of Cellscope makes mocking non-trivial as it can span across microservice boundaries.

Machine Learning

Verifying the correctness of machine learning tasks and scanning large output files for correctness is an area of active research by CERN, Alphabet, Uber, etc.

Long Runtimes

End-to-end tests are bottlenecked by machine learning tasks that can take hours to run on larger data sets.

Objective, Tools and Practices

Objective

The objective is to make continuous delivery possible for Cellscope. Ideally, all commits to the mainline branch should be ran against the entire suite of end-to-end tests, and the tests should complete in a reasonable amount of time while providing the necessary information about the current state of the product and whether or not it is release-ready.

Practice: Mocking

Mocking is very likely going to be necessary. The correctness of most machine learning algorithms is usually verified by the package creator in their unit tests. However, mocking out R packages will prove challenging. In addition, mocking is not straightforward as the platform is distributed and will take careful engineering to accomplish well.

Tools: Regression Testing

It should be possible to ensure that changes are only run against end-to-end that are likely to be affected by the change. Creating a change set as a regression test suite appropriately is not trivial and can present many edge cases.

Tools: Reworking Test Hierarchies

Some large-scale projects take this approach. The entire system is broken up into large, testable chunks comprising multiple services, each of which is tested using their own integration tests, with connections to other services mocked out. If the communication protocol is well-established, one can then separately test the validity of the interfaces as a form of end-to-end testing.
